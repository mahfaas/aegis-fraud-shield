# Aegis Fraud-Shield: Максимально подробный разбор проекта

Добро пожаловать в полное руководство по архитектуре, фичам и реализации **Aegis Fraud-Shield**! Этот документ написан специально для вас, чтобы вы досконально понимали, как работает каждая деталь созданной нами системы, могли свободно в ней ориентироваться и защищать её архитектуру.

---

## 1. Что из себя представляет проект?

**Aegis Fraud-Shield** — это высоконагруженный (high-load), антифрод-движок реального времени.
Его задача: непрерывно читать поток финансовых транзакций (переводы, покупки), "на лету" пропускать их через каскад проверок (правил) и мгновенно выносить вердикт: пропустить платеж, отклонить как мошеннический или отправить на ручную проверку.

### Ключевые фичи (Что он умеет):
1. **Обработка потоков (Event-Driven)**: Мы слушаем топик Kafka. Никакого медленного поллинга баз данных — транзакция влетает, проверяется за миллисекунды, и результат улетает дальше в другой топик.
2. **Паттерн "Цепочка обязанностей" (Chain of Responsibility)**: Правила работают как конвейер. Если одно правило сразу говорит "ФРОД!", остальные даже не запускаются (экономим вычислительные ресурсы).
3. **Окна времени и Скорость (Velocity)**: Встроенный Redis отслеживает количество транзакций от одного пользователя в секунду/минуту (чтобы остановить брутфорс или угон карты).
> *"Антифрод-система должна в первую очередь быть неблокирующей и дебаунснутой. Я выбрал Kafka, потому что она сглаживает пики (spikes) трафика — даже если придет 1 миллион транзакций, сервер не упадет с ошибкой 500, он просто сложит их в топик и обработает с максимально доступной ему скоростью. Я использовал Redis для stateful-правил (состояния), потому что SQL база данных не выдержит UPDATE'ов и SELECT'ов при пересчете каждой транзакции в реальном времени. А для статических правил я применил In-memory кэш на Concurrent Set'ах — это классический прием для высокочастотного трейдинга и финтеха (Low Latency), сводящий время ответа к минимуму."*
4. **Геолокационные аномалии (Impossible Travel)**: Тот же Redis помнит, откуда была прошлая транзакция. Если 5 минут назад вы платили из `RU`, а сейчас из `US` — это физически невозможное путешествие, аккаунт блокируется.
5. **Динамический Черный Список (In-memory + DB)**: Заблокированные IP и номера карт (BIN) хранятся в PostgreSQL для надежности, но копируются в оперативную память (RAM) приложения. Благодаря этому проверка занимает O(1) (мгновенно), без необходимости делать долгий `SELECT` в базу каждый раз!
6. **Управление порогами "На Горячую"**: Через REST API можно поменять пороги сумм или лимиты скорости без перезагрузки приложения!
7. **Мониторинг (Observability)**: Каждая операция автоматически пишет статистику в Micrometer, которую забирает Prometheus и рисует Grafana (дашборды).

### Стек технологий:
- **Core**: Java 21, Spring Boot 3.4
- **Messaging**: Apache Kafka (быстрая передача транзакций)
- **State/Cache**: Redis (алгоритмы скользящего окна времени, геолокация)
- **Database**: PostgreSQL (хранение настроек и блэклиста), Spring Data JPA, Flyway (миграции БД)
- **Monitoring**: Micrometer, Prometheus, Grafana

## 2. Жизненный цикл транзакции (Как бегут данные)

1. Новая транзакция (JSON) попадает в Kafka-топик `transactions-raw`.
2. `TransactionConsumer` считывает её.
3. Отрабатывает `TransactionValidator` (проверяет, есть ли вообще сумма, не пустой ли ID).
4. Запускается `RuleEngine` (Движок правил).
5. Движок по очереди пропускает транзакцию через 4 правила.
6. Выносится вердикт: `APPROVED`, `DECLINED` или `MANUAL_REVIEW`.
7. `VerdictProducer` отправляет этот вердикт в Kafka-топик `transactions-verdicted`.

## 3. Детальный разбор кода (Как это реализовано под капотом)

Давайте залезем в код. Я разберу самые важные и сложные места "чуть ли не построчно".

### Блок A. Потребление сообщений (Kafka)

#### `io.github.mahfaas.fraudshield.kafka.TransactionConsumer`
Это наш "вход в приложение".

```java
@KafkaListener(topics = "${fraud.kafka.topic-in}", groupId = "${spring.kafka.consumer.group-id}")
public void consume(Transaction transaction) {
    // 1. Отмечаем в метриках (для Grafana), что получили новую транзакцию
    metrics.recordReceived();

    // 2. Валидация (просто проверяем поля, например, что amount > 0)
    List<String> validationErrors = validator.validate(transaction);
    if (!validationErrors.isEmpty()) {
        // Если данные битые, шлём их в DLQ (Dead Letter Queue - топик для "мусора" и ошибок)
        verdictProducer.sendToDlq(transaction.getTransactionId(), ...);
        metrics.recordDlq(); 
        return; // Завершаем обработку!
    }

    // 3. Запускаем "Движок правил" и замеряем время выполнения (startTimer)
    Timer.Sample timer = metrics.startTimer();
    VerdictedTransaction result = ruleEngine.evaluate(transaction); // СЕРДЦЕ СИСТЕМЫ
    metrics.stopTimer(timer);

    // 4. Отправляем результат дальше в выходной топик Kafka
    verdictProducer.send(result);
    metrics.recordVerdict(result.getVerdict());
}
```

### Блок B. Движок правил (RuleEngine - Цепочка обязанностей)

Класс `RuleEngine` — дирижер. Ему Спринг автоматически "впрыскивает" (inject) список всех классов, которые реализуют интерфейс `Rule`.

```java
public VerdictedTransaction evaluate(Transaction transaction) {
    List<String> reasons = new ArrayList<>();
    Verdict finalVerdict = Verdict.APPROVED;

    // Идем по списку всех активных правил (Blacklist -> Amount -> Velocity -> GeoVelocity)
    for (Rule rule : rules) {
        // Запускаем очередное правило
        RuleResult result = rule.evaluate(transaction);

        // Практика "Short-Circuit" (Короткое замыкание)
        if (result.getVerdict() == Verdict.DECLINED) {
            reasons.add("[" + rule.getName() + "] " + result.getReason());
            finalVerdict = Verdict.DECLINED;
            break; // Если транзакция отклонена, НАМ НЕ НУЖНО ПРОВЕРЯТЬ ОСТАЛЬНЫЕ ПРАВИЛА! Выходим из цикла.
        }

        // Если правило говорит "Подозрительно, нужен ручной обзор"
        if (result.getVerdict() == Verdict.MANUAL_REVIEW) {
            reasons.add("[" + rule.getName() + "] " + result.getReason());
            finalVerdict = Verdict.MANUAL_REVIEW;
            // Здесь НЕ делаем break! Почему? Потому что следующее правило 
            // может сказать DECLINED (строгий запрет), что приоритетнее.
        }
    }
    // Формируем финальный объект с вердиктом и причинами
    return VerdictedTransaction.builder()...build();
}
```

### Блок C. Конкретные правила (Heart of Fraud Detection)

#### 1. `BlacklistRule` (Умный кэш)
Это правило проверяет черные списки IP и номеров карт (BIN). Но мы не можем делать `SELECT` в базу для каждой транзакции — база ляжет (зависнет), если придет 10,000 запросов в секунду. Поэтому используется **In-memory cache**.

У нас есть два потокобезопасных поля:
```java
private final Set<String> blacklistedIps = ConcurrentHashMap.newKeySet();
private final Set<String> blacklistedBins = ConcurrentHashMap.newKeySet();
```
При запуске приложения `BlacklistLoader` читает базу ПОЛНОСТЬЮ и закидывает все в эти два сета.
Сама проверка выглядит просто:
```java
// O(1) скорость проверки — поиск в хэш-сете происходит мгновенно
if (blacklistedIps.contains(transaction.getSourceIp())) {
    return RuleResult.decline(RULE_NAME, "IP is blacklisted");
}
```
*Интересная деталь*: когда вы через API ( POST /api/v1/blacklist) добавляете новый IP, контроллер сохраняет его в БД и **одновременно** кладёт в этот Set, чтобы `BlacklistRule` сразу узнал о новом заблокированном IP без перезагрузки!

#### 2. `VelocityRule` (Атака перебором)
Как понять, что с 1-го аккаунта делают 100 платежей в минуту? Нужна система, которая помнит "состояние", причем распределенное (если у нас запущено 5 инстансов приложения). Для этого мы берем **Redis**.

Мы используем команды Redis: `INCR` (Увеличить на 1) и `EXPIRE` (Установить время жизни).
```java
// Ключ в Redis будет выглядеть так: "velocity:ACC-001"
String key = KEY_PREFIX + transaction.getAccountId();

// Делаем +1. Redis делает это атомарно и сверхбыстро (за доли миллисекунды)
Long count = redisTemplate.opsForValue().increment(key);

// Если это первая транзакция в этом окне времени, ставим таймер на удаление
// Например, 60 секунд. Через минуту этот ключ из Redis самоликвидируется!
if (count != null && count == 1) {
    redisTemplate.expire(key, Duration.ofSeconds(windowSeconds));
}

// Проверяем: если счетчик стал 6 (а лимит 5) — БЛОКИРУЕМ!
if (count != null && count > maxTransactions) {
    return RuleResult.decline(RULE_NAME, "Velocity exceeded: " + count + " txs in " + windowSeconds + "s");
}
```
Это гениальный паттерн **Sliding Window Counter** на базе Redis!

#### 3. `GeoVelocityRule` (Невозможное путешествие)
Тоже использует Redis, но вместо счетчика мы храним строку: `Страна|Время_последней_транзакции`.

```java
String key = "geo:" + transaction.getAccountId();
// Читаем, где он был в прошлый раз
String storedValue = redisTemplate.opsForValue().get(key);

// Разделяем Страну и Время
String[] parts = storedValue.split("\\|");
String previousCountry = parts[0];
long previousTimestamp = Long.parseLong(parts[1]);

// Смотрим, сколько секунд прошло с прошлого раза
long elapsedSeconds = (текущее_время) - previousTimestamp;

// Если ПРОШЛАЯ страна не совпадает с ТЕКУЩЕЙ и прошло МЕНЬШЕ часа (<3600с)
if (!previousCountry.equals(currentCountry) && elapsedSeconds < WINDOW_SECONDS) {
    // "Невозможное путешествие" — человек не мог долететь из США в Россию за 5 минут!
    return RuleResult.manualReview(...);
}
```

#### 4. `AmountAnomalyRule` (Динамические пороги)
Самое простое правило. Если сумма > X — Decline, если сумма > Y — Manual Review.
Но главная его фича — мы не "зашиваем" X и Y жестко в код.
```java
// Поля НЕ marked as 'final'
private BigDecimal declineThreshold;
private BigDecimal reviewThreshold;
```
Через REST-контроллер `RuleConfigController` мы можем в рантайме (пока приложение работает) послать JSON и обновить эти пороги с помощью методов `setDeclineThreshold()`. При следующей транзакции правило использует уже новые лимиты! Это называется **Runtime Configuration Update**.

### Блок D. Генератор нагрузки и баг, который мы починили (Producer)

Как работает тестовые данные? У нас есть `TransactionProducer.java`.
Там написан алгоритм, который генерирует "фейковые" транзакции в точных пропорциях с помощью функции `random.nextDouble()` (если < 0.70 — нормальная, если от 0.7 до 0.8 — velocity, и т.д.).

**Какой баг был до этого?**
У генератора была жестко зашитая база из 10 аккаунтов (`ACC-001`, `ACC-002`... `ACC-010`). Когда генератор создавал 500 "нормальных" транзакций, он распределял их случайно между этими 10 аккаунтами. В среднем на каждый выпадало по 50 транзакций мгновенно. 
Что делало `VelocityRule`? Оно видело 50 транзакций за 1 секунду с одного аккаунта `ACC-005` и блокировало всё, начиная с 6-ой! В итоге 86% транзакций гибло.
*Решение*: Я изменил код `buildNormal()` — теперь он использует `ACC-NORM-{Random от 1 до 100_000}`. Огромное пространство имен аккаунтов означает, что нормальные транзакции никогда не пересекутся и не сработают на счетчике Velocity, а вот `buildVelocityAnomaly()` бьет прицельно в один конкретный счётчик `ACC-VELOCITY`, чтобы специально триггерить аномалию!

## 4. Как это выглядит для Бизнеса и DevOps

Система покрыта метриками Micrometer. Что это значит программистским языком?
Класс `FraudMetrics` хранит объекты `Counter` (счётчик) от библиотеки Micrometer.
Где бы мы ни вызвали `counter.increment()`, Micrometer прибавляет +1 в памяти.

Далее Spring Actuator обнажает URL: `http://localhost:8080/actuator/prometheus`.
Там лежат сырые текстовые метрики.
Docker-контейнер **Prometheus** (база данных для временных рядов) раз в 15 секунд стучится по этому URL и забирает все метрики себе в хранилище.
Docker-контейнер **Grafana** (система дашбордов) подключается к Prometheus, выполняет запросы на языке PromQL (например, `rate(fraud_transactions_received_total[1m])` — количество транзакций в минуту) и красиво рисует это в виде графиков и спидометров на вашем экране.

## Итог (Summary)

Aegis Fraud-Shield — это взрослая, микросервисная архитектура. 
Она разделяет потоки: `Kafka` отвечает за очередь и консистентность, `Redis` — за быстрые математические окна (velocity rules), `Postgres` — за долгое хранение списков, а сама `Java` — чистое ядро бизнес-логики (RuleEngine).
Тестирование (49 Unit-тестов, написанных с помощью моков Mockito) гарантирует, что любое изменение в логике (например, в подсчете скорости) мгновенно подсветит ошибку при компиляции.

## 5. Микросервисы, Docker и Магия Запуска

### А это микросервисы или нет?
**Aegis Fraud-Shield** сам по себе (наш Java-код) является одним **микросервисом** (или как это стильно называть — "Модульным Монолитом", потому что весь наш антифрод находится в одном JAR-файле).
Почему мы не разбили его на 10 разных Java-проектов? Потому что отправка данных по сети между мелкими сервисами занимает драгоценные миллисекунды. Антифрод требует оценки "здесь и сейчас" внутри одного куска памяти.

Однако, **вся наша инфраструктура** развернута в глубоко микросервисном подходе. Взгляните на файл `docker-compose.yml`, который поднимает 6 ИЗОЛИРОВАННЫХ контейнеров (сервисов), каждый из которых выполняет строго одну задачу.

### Что крутится в Docker-контейнерах?
Когда вы пишете `docker-compose up -d`, создается локальная сеть Docker (мост Bridge), в которой поселяются 6 независимых "мини-компьютеров":

1. **`fraud_postgres` (PostgreSQL 15)**: Реляционная база. Она гарантирует консистентность конфигураций и черных списков (ACID). Если свет выключат, списки блокировок не пропадут.
2. **`fraud_redis` (Redis 7)**: NoSQL база данных в оперативной памяти. В ней живут счетчики транзакций (Velocity) и страны с таймерами (`EXPIRE`). Разрыв питания удалит эти данные, и это нормально для *скользящих ограничений скорости*, которые актуальны только здесь и сейчас!
3. **`fraud_zookeeper`**: Дирижер для Kafka. Он знает, какие брокеры Кафки живы, а какие нет.
4. **`fraud_kafka` (Apache Kafka 7.5)**: Брокер сообщений. Это "сердце" распределенных систем. Кафка сохраняет поток транзакций на диск (append-only log), поэтому даже если наше Java-приложение остановится на 10 минут, транзакции накопятся в Kafka, и когда приложение оживет — оно выпьет всю очередь без потери единого платежа.
5. **`fraud_prometheus`**: Time-Series база данных. Сервис-шпион, который каждые 15 секунд стучится к нашему Java-приложению по порту 8080 (URL: `/actuator/prometheus`) и выкачивает метрики (сколько транзакций было, сколько заблокировано).
6. **`fraud_grafana`**: Веб-интерфейс визуализации. Сама данные она не хранит. Она делает SQL-подобные запросы (PromQL) в Prometheus и рисует красивые дашборды, на которые смотрят бизнес-аналитики.

### Как собирается и запускается Спринг (Что происходит в консоли при старте)
Когда вы вводите команду `./mvnw spring-boot:run` или запускаете собранный `.jar` файл, происходит настоящая магия загрузки:

1. **Создание Контекста (DI/IoC)**: Сперва Spring сканирует все папки и создает `Singleton Bean`-ы (одиночные инстансы классов, аннотированных `@Component`, `@Service`). Он выстраивает граф зависимостей.
2. **Flyway (Миграции Базы Данных)**: До того как разрешить приложению работать, библиотека Flyway подключается к PostgreSQL. Спринг ждет. Flyway смотрит в служебную таблицу `flyway_schema_history` и видит: *"Ага, база пустая. Применю-ка скрипт `V1__init_schema.sql`"*. Она прогоняет скрипт и создает таблицы `blacklist_entry`.
3. **Hibernate (ORM)**: Далее Spring Data JPA проверяет ваши `Entity` (например, `BlacklistEntity`) и связывает (маппит) их с таблицами, которые только что заботливо создал Flyway. Настраивается пул соединений к базе (`HikariCP`).
4. **Загрузка в ОЗУ (BlacklistLoader)**: Вызывается наш загрузчик. Он делает SQL запрос: `SELECT * FROM blacklist_entry` и наполняет потокобезопасные `ConcurrentHashMap` внутри `BlacklistRule`, чтобы всё было готово к мгновенным проверкам в оперативной памяти.
5. **Подключение к брокеру (Kafka Consumer Rebalance)**: 
   И только в самом конце (когда все внутренности готовы и базы прогреты), Spring говорит Кафке: *"Я готов принимать данные!"*. 
   В логах вы видите `(Re-)joining group`. Наш `TransactionConsumer` логинится под именем `fraud-engine-group`, и Кафка поручает ему чтение партиций (разделов) из топика `transactions-raw`. 
6. **Tomcat стартовал**: В логе появляется `Tomcat started on port 8080`. Теперь мы можем посылать REST HTTP запросы на наши контроллеры! Система вошла в рабочий режим!
